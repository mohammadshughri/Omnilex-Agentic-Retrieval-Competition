02 — The Problem

The Cross-Lingual Gap

Training queries are German. Validation and test queries are English. The entire corpus — 176K law articles and 2.4M court decisions — is German/French/Italian.

Why BM25 fails: BM25 scores documents by word overlap. English "detention" has zero overlap with German "Haft". For conceptual queries with no explicit article numbers, BM25 returns nothing useful. Our baseline Macro F1 on English queries via BM25: approximately 0.02.

03 — Key Concepts

Key Concepts

Click any concept to expand the full explanation.

BM25 — keyword ranking (the baseline we're replacing)

BM25 (Best Match 25) is a classic TF-IDF-based ranking algorithm. It scores documents by counting how often query tokens appear in the document, normalized by document length. It's fast, interpretable, and works well monolingually.

Why it fails here: BM25 relies on exact token overlap. English "detention" shares zero characters with German "Haft". For queries that describe a legal situation without mentioning article numbers, BM25 returns a score of 0 for most documents.

Where BM25 still helps: Queries that explicitly mention Swiss law identifiers like Art. 221 StPO — those tokens appear literally in the corpus and BM25 finds exact matches. We use BM25 as an additional channel via RRF fusion for this reason.

Dense Embeddings — semantic search across languages

A transformer model encodes any text into a fixed-size vector (e.g., 1024 dimensions). Semantically similar texts — even in different languages — land near each other in this vector space.

We pre-compute embeddings for the entire corpus (176K laws + 2.4M court passages) and store them in a FAISS index. At query time, we embed the English question and retrieve the nearest vectors in milliseconds.

Why this solves the cross-lingual gap: A model trained on multilingual data maps "detention" and "Haft" to similar locations in embedding space, even though they share no tokens.

BGE-M3 — three retrieval signals from one model

BAAI/bge-m3 is a 0.6B parameter model that produces three complementary retrieval signals from a single forward pass:

  • Dense embeddings (1024-dim) — semantic similarity, works cross-lingually
  • Sparse neural weights — cross-lingual neural "BM25", no vocabulary mismatch
  • ColBERT multi-vector — one vector per token per passage, precise but expensive (laws only)

BGE-M3 fits on 8GB VRAM. It is our Week 1-2 retriever. Qwen3-Embedding-8B (MTEB #1 multilingual) is planned for Week 5-6.

We combine the three signals via RRF fusion — not weighted score addition. See the RRF concept card below for why.

FAISS — fast approximate nearest-neighbor search at 2M+ scale

FAISS (Facebook AI Similarity Search) is a library for efficient similarity search over dense vectors. Doing an exact nearest-neighbor search over 2.4M vectors naively would be too slow.

We use IndexHNSWFlat (Hierarchical Navigable Small World, M=32): ~98% recall vs. exact search, no training step required, query time in milliseconds.

Memory estimates: Laws FAISS index ~500MB; Courts FAISS index ~7-8GB. The T4 on Kaggle has 16GB VRAM — we load laws to GPU and courts to CPU.

We build two separate indices (laws and courts) rather than one combined index — see the Architecture section for the reasoning.

RRF — Reciprocal Rank Fusion (how we combine retrieval channels)

RRF combines multiple ranked lists by position, not by score. Formula:

score(doc) = ∑ over each channel: 1 / (60 + rank_in_that_channel)

Why not add scores directly? Dense, sparse, BM25, law, and court scores are on completely different numeric scales. Adding them directly produces nonsense rankings. RRF only uses the rank position, which is always 1-to-N and directly comparable.

RRF was adopted after an architecture review rejected an earlier weighted-score approach. Do not reintroduce weighted scores without strong calibration evidence.

HyDE — Hypothetical Document Embeddings (Week 4)

Instead of embedding the English query directly, we use an LLM to generate a hypothetical German law passage that would answer the query, then embed that synthetic passage and search the corpus with it.

Why this helps: The synthetic passage is in German (the corpus language) and uses legal terminology. Its embedding naturally lands near actual German law passages, bridging the cross-lingual gap without explicit translation.

Limitations: If the LLM hallucinates incorrect article numbers in the synthetic passage, those wrong numbers pull retrieval toward wrong documents. We use the original English query embedding at the reranker step, not the synthetic passage.

HyDE is deferred to Week 4 — we want a stable dense retrieval baseline first.

Macro F1 — the competition metric

Macro F1 averages the F1 score per query, giving every query equal weight regardless of how many citations it has. F1 = 2 × (precision × recall) / (precision + recall).

Implication for prediction: A query with 1 gold citation and a query with 44 gold citations contribute equally to the final score. Predicting too many citations hurts precision; predicting too few hurts recall. This is why predicting the right number of citations matters as much as which ones.

Current top leaderboard score: 0.359. Our BM25 baseline on English queries: ~0.02. Target with BGE-M3 hybrid: 0.15–0.25.

Reranker — precision over top-100 candidates (Week 3+)

First-stage retrieval (FAISS) optimizes recall — get all relevant documents in the top-100. A reranker then optimizes precision — score each (query, document) pair carefully and reorder.

A reranker is a cross-encoder: it sees both the query and document at once (unlike a bi-encoder which embeds them independently). More accurate but 100× slower — so it runs on top-100, not the full 2M+ corpus.

We use bge-reranker-v2-m3 in Week 3, then upgrade to Qwen3-Reranker-8B in Week 5-6. Note: Qwen3-Reranker must be loaded as a causal LM and scored via yes/no token logits — not a standard CrossEncoder API.

04 — Architecture

Pipeline Architecture

English Legal Query │ ├──► Citation Anchor Extractor ← regex: "Art. 221 StPO" found literally in query │ └──► CitationNormalizer ← canonical form (existing module) │ └──► High-priority candidates (injected at rank 0 in fusion) │ ├──► BGE-M3 Encoder (0.6B params, BAAI/bge-m3) │ ├── Dense embeddings (1024-dim) │ ├── Sparse neural weights (cross-lingual lexical) │ └── ColBERT tokens ← laws only (175K); too expensive for 2.4M courts │ ▼ FAISS Index (pre-built, loaded from disk) ├── Laws Index (175K passages) ~500 MB └── Courts Index (2.4M passages) ~7-8 GB [dense+sparse only] │ ▼ Top-100 candidates per channel (dense / sparse / ColBERT each ranked independently) │ ▼ RRF Fusion (per corpus) score = Σ 1/(60 + rank) ← rank-based; no score-scale issues │ ▼ Federated Law + Court Merge deduplicate by normalized citation │ ▼ Reranker: bge-reranker-v2-m3 ← Week 3+ │ ▼ Dynamic K / Score Threshold ← test k=2,3,5,10 vs threshold on val │ ▼ CitationNormalizer → submission.csv
⚠️ Top 3 Silent Failure Modes
  1. Weighted scores instead of RRF. Dense, sparse, BM25, law, and court scores are on incompatible scales. Adding them directly produces nonsense rankings. Always use RRF.
  2. Comparing raw citation strings without CitationNormalizer. "Art. 1 Abs. 1 ZGB" and "Art. 1 ZGB" are the same citation but won't match as strings. Always normalize before comparing. F1 silently underreports if you skip this.
  3. Tuning hyperparameters against val.csv more than once. val.csv has only 10 English queries. Repeated tuning is effectively memorization, not generalization. Use it as a final sanity check, not an optimization target.

Why Two Separate Indices?

Laws Index

175K passages · ~500 MB · German only · Short snippets (~242 chars) · Build time: ~20 min

Courts Index

2.4M passages · ~7-8 GB · DE/FR/IT · Long passages (~1,105 chars) · Build time: 4-8 hrs on GPU

Separate indices allow failure isolation, different memory strategies (laws on GPU, courts on CPU), and separate weighting matching the 70% laws / 30% courts split in gold labels. ColBERT multi-vector is only applied to the laws corpus — 2.4M courts passages would exceed memory.

Architecture Evolution: v1 → v2

v1 (rejected): Weighted score fusion: 0.4×dense + 0.2×sparse + 0.4×ColBERT. An architecture review found these scores are on incompatible numeric scales — adding them is undefined.

v2 (current): RRF replaces weighted scores everywhere. Do not reintroduce score weighting without calibration data showing it beats RRF on val.csv.

Model Pluggability

All model calls go through protocol interfaces — swapping models requires only a new adapter class:

# EmbeddingModel protocol — encode_documents() / encode_queries()
# BgeM3Embedder implements it (Week 1-2)
# Qwen3Embedder will implement it (Week 5-6) — no retrieval code changes needed

# RerankerModel protocol — rerank(query, docs)
# BgeCrossEncoderReranker (Week 3)
# Qwen3CausalLmReranker (Week 5-6)
#   Note: Qwen3-Reranker uses yes/no token logits — NOT the standard CrossEncoder API

09 — Rules

Rules & Conventions

Package Management

Always use uv pip install — never plain pip install. Reproducible lockfiles, faster resolution.

Install the local package in dev mode: uv pip install -e .

Running Code

source .venv/bin/activate     # activate first
pytest                         # run before pushing
ruff check . && ruff format . # lint before pushing

Evaluation Discipline

⚠️ val.csv has only 10 queries. Do NOT tune hyperparameters against it repeatedly. Ten queries is statistically meaningless — repeated tuning is memorization, not generalization. Use it as a final sanity check only. Report val.csv scores as "calibration check", not strong evidence. Use train-derived diagnostics for iteration — but label them as German-query diagnostics (different language from test).

Data Rules

  • Never commit raw data — all data paths are gitignored
  • data/raw/ is read-only — never modify source files
  • Processed files (embeddings, FAISS indices) go in data/processed/ (gitignored, ~15GB)

Artifact Safety

⚠️ Do NOT use pickle for shared or uploaded artifacts. Use .npy, FAISS native format, or JSON instead. The existing laws_index.pkl / courts_index.pkl are acceptable for local BM25 prototyping only. Pickle is unsafe for untrusted artifacts and impractical at 2.4M passages.

Citation Normalization

Always pass raw citation strings through CitationNormalizer before comparing or submitting.

from omnilex.citations.normalizer import CitationNormalizer
normalizer = CitationNormalizer()
# "Art. 1 Abs. 1 ZGB" → same normalized form as "Art. 1 ZGB"
canonical = normalizer.normalize("Art. 1 Abs. 1 ZGB")

Never write alternative regex parsers for citations — CitationNormalizer is the single canonical parser.

Do Not Reintroduce

  • Raw weighted score fusion — use RRF instead; scores are not cross-scale comparable
  • Hard-coded model paths — use the EmbeddingModel / RerankerModel protocol so Qwen3 is a clean swap-in
